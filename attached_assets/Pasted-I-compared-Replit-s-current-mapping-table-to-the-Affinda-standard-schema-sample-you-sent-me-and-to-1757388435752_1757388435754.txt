I compared Replit’s current mapping table to the Affinda “standard schema” sample you sent me and to the mapper I gave you. Here are the concrete gaps and how to fix them so everything lines up and we stop dropping fields.

What’s off right now (and why)

Missing data. root

Replit’s paths start at candidateName, email, etc. The Affinda doc you sent is rooted at data.*.

Fix: All selectors should address raw.data first. If your adapter already does const d = raw?.data || raw, you’re fine; if not, add it or prepend data. to every path.

Heavy reliance on .parsed nodes

Replit expects candidateName.parsed.firstName.parsed, workExperience[n].parsed.*, education[n].parsed.*.

The standard sample shows values directly on the element (e.g., workExperienceOrganization, educationAccreditation) and not under .parsed.

Fix: Try .parsed first, then fall back to the direct field (no .parsed), then legacy names.

Arrays vs. object-with-numeric-keys

Replit notes work/education/skills may be objects like {'0': {...}, '1': {...}}.

The sample uses listItem[] arrays.

Fix: Always normalize collections with const arr = Array.isArray(x) ? x : Object.values(x || {}).

Email/phone “shape”

Replit uses email.listItem[0].value and phoneNumber.listItem[0].formattedNumber.

The sample often has listItem strings (no .value) and phone may expose .rawText.

Fix: For email: try .listItem[0].value → .listItem[0]. For phone: try .formattedNumber → .rawText → raw string. Keep regex fallback as last resort.

Location fields

Replit favors location.parsed.*; the sample is location.formatted, location.city, location.state/stateCode, location.postalCode, location.countryCode.

Fix: Try location.parsed.* first, then the non-parsed keys. Prefer countryCode (ISO) over country for consistency.

Work experience keys

Replit: workExperience[n].parsed.workExperienceOrganization, .workExperienceJobTitle, .workExperienceStartDate.parsed, .workExperienceEndDate.parsed, .workExperienceLocation.parsed, .workExperienceType.value.

Sample: the same field names but inside each list item (no .parsed) and dates under workExperienceDates.start.date / end.date (+ end.isCurrent).

Fix: Selector order per field should be parsed → Dates.start.date/Dates.end.date → startDate/endDate → dates.start/dates.end.

Education keys

Replit: education[n].parsed.educationOrganization, .educationAccreditation, .educationMajor.listItem[0], .educationStartDate.parsed, .educationEndDate.parsed, .educationLocation.formatted.

Sample: same names as above but dates under educationDates.start.date/end.date; major array under educationMajor.listItem.

Fix: Use parsed first, then educationDates.*.date, then startDate/endDate, then dates.start/dates.end.

Skills / keywords

Replit: skill[n].parsed.skillName with fallback to skill.listItem[n].name; keywords from keywordSet.listItem[n].

Sample: data.skill.listItem[].name.

Fix: Keep both; if keywordSet exists, map to your optional “keywords” field, otherwise ignore.

LinkedIn

Replit: socialMedia.listItem[].url (LinkedIn) with fallbacks.

You’ll want to filter by provider and then pick the URL.

Fix: Select the list item where platform/name contains “LinkedIn”, then take .url.

Current job

Replit: currentJobTitle.parsed fallback workExperience[0].position.

Solid; just ensure you pick the current work item based on end.isCurrent=true rather than assuming index 0.

Code-level adjustments (so the table actually works)

Normalize collections everywhere:

const norm = (x) => Array.isArray(x) ? x : Object.values(x || {});
const work = norm(d.workExperience?.listItem ?? d.workExperience ?? d.experience ?? d.positions);
const edu  = norm(d.education?.listItem ?? d.education ?? d.educationHistory);
const skl  = norm(d.skill?.listItem ?? d.skill ?? d.skills);


Try .parsed → direct → legacy: always in that order.

Date normalization: run normDate(YYYY-MM-DD|YYYY-MM|YYYY) to YYYY-MM; set endDate = null if isCurrent=true.

Regex fallbacks for email/phone if structured paths are empty.

ISO country: if both country and countryCode present, store the code and keep country as display-only if you want.

Prefer the “current job” using isCurrent instead of index order.

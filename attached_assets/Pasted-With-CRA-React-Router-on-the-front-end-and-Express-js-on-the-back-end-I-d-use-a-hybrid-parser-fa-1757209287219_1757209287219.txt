With CRA + React Router on the front end and Express.js on the back end, I’d use a hybrid parser: fast, deterministic extraction first, then an LLM or rules-based mapper to normalize into a clean schema (JSON Resume). Here’s the approach I recommend.

TL;DR stack

Upload & validation: multer (Express) for file upload; allow PDF/DOCX only (flag others).

Extraction (deterministic first):

PDF: pdf-parse (text-layer PDFs) → fast, reliable.

DOCX: mammoth (to HTML/text) → preserves logical order/headings.

Fallbacks: textract as a catch-all for odd files; OCR via tesseract.js (Node) or an OCR service when we detect “image-only PDF.”

Segmentation & cleanup: cheerio to parse DOCX→HTML; simple header heuristics (e.g., ^experience$, ^education$, ^skills$), bullet normalization, de-hyphenation across line wraps.

Field extraction helpers (Node):

Email: email-validator

Phone: libphonenumber-js

Dates: chrono-node + date-fns normalize to YYYY-MM

Locations: light heuristics now; can add node-postal later if you want global address parsing

Schema & validation: JSON Resume as the canonical structure; enforce with zod.

Intelligence layer (only where needed):

If headings are unclear or the resume is atypical, pass the plain text through an LLM with a strict JSON schema (function/tool calling) to map to JSON Resume. Keep PII server-side.

Confidence scoring: per field (high/med/low). Use confidence to highlight fields in Steps 3–6 for user confirmation.

Security & ops: Virus scan (optional ClamAV), size/type limits, store in private bucket, encrypt at rest, signed URLs, redact SSN to last 4, log access.

Data flow (how it plugs into your onboarding)

Step 2 (Upload Résumé) → POST /api/resume/parse

Express flow:

Validate file (size/MIME).

Detect has text layer? If no → OCR; if yes → parser path.

Extract raw text + simple structure (sections, bullets).

Run deterministic extractors (contact, education, experience, skills).

If coverage is low or headings are ambiguous → AI mapper to JSON Resume.

Normalize & validate with zod; compute confidence per field.

Response: { jsonResume, prefill: { step3, step4, step5, step6 }, confidence }

Client (Step 3–6): prefill fields; badge low-confidence ones for review.

JSON shape (what we store)

Primary: JSON Resume (portable, works for future exports).

Derived for your UI:

Why this works well for Talendro™

Fast for the 80% case (standard PDFs/DOCX), but robust when resumes are quirky (LLM mapper & OCR fallback).

Deterministic core (you can unit test it) + LLM just for mapping (bounded, schema-validated).

Portable storage (JSON Resume) that feeds your Step 3–6 forms and future exports.

Confidence-driven UX reduces user friction: only review what’s uncertain.

Nice-to-haves (v2)

Skill ontology + synonym map (e.g., “PMP” ⇄ “Project Management Professional”) using a small dictionary + fuzzy matching (Fuse.js) and/or embeddings for smarter grouping.

Two-column PDF reading order fixer (simple heuristics: column gap detection, reflow).

Background OCR queue for huge scans so Step 2 isn’t blocked (return “Parsing… we’ll fill as soon as ready,” but still let them proceed).
